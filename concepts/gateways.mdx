A Gateway in N-Gate represents an upstream connected service — an institution, a partner system, or any backend you want to make accessible in a controlled, authenticated way. Gateway Consumers are the API clients and applications that have been explicitly authorized to send requests through a specific gateway. Together, these two concepts give you precise control over who can call what, on behalf of which upstream system, without exposing raw credentials or direct backend access to anyone.

## The Gateway Model

Think of a Gateway as a named, managed connection to an upstream service. When you create a gateway record in N-Gate, you register the upstream service's identity and connection details — N-Gate then acts as the authenticated intermediary for every call that flows through it.

```text
┌──────────────────────────────────────────────────────────────┐
│                         N-GATE                               │
│                                                              │
│  ┌─────────────────────┐       ┌──────────────────────────┐  │
│  │  GATEWAY CONSUMER   │──────▶│        GATEWAY           │  │
│  │                     │  authorized  │                   │  │
│  │  API Client / App   │       │  Upstream Service Record │  │
│  │  (Bearer token)     │       │  (institution / partner) │  │
│  └─────────────────────┘       └────────────┬─────────────┘  │
│                                             │                │
└─────────────────────────────────────────────┼────────────────┘
                                              │ authenticated proxy
                                              ▼
                                 ┌────────────────────────┐
                                 │   UPSTREAM SERVICE     │
                                 │  (partner API, backend │
                                 │   institution, etc.)   │
                                 └────────────────────────┘
```

A request from a Gateway Consumer only reaches the upstream service if:
1. The consumer is registered and linked to the target gateway.
2. The caller's Bearer token is valid.
3. The N-Gate authorization layer approves the request against the workspace's Permission Roles.

<Note>
  Gateways are scoped to a workspace. A gateway created in one workspace is invisible to all other workspaces, maintaining full multi-tenant isolation. All gateway and consumer endpoints require a valid `Authorization: Bearer <token>` header.
</Note>

## Gateways vs. Gateway Consumers

<CardGroup cols={2}>
  <Card title="Gateway" icon="tower-broadcast">
    Represents an upstream service or institution. Stores the connection configuration, upstream endpoint details, and any service-level credentials. Administered at `/api/gateways`.
  </Card>
  <Card title="Gateway Consumer" icon="user-check">
    Represents an API client authorized to call through a specific gateway. Holds the consumer's identity and the list of gateways they are permitted to use. Managed at `/api/gateway-consumer`.
  </Card>
</CardGroup>

The relationship is many-to-many: a single gateway can serve multiple consumers, and a single consumer can be authorized to call through multiple gateways. You manage these links explicitly, so access is always intentional and auditable.

## Administering Gateways

<Tabs>
  <Tab title="Create a Gateway">
    <Steps>
      <Step title="Register the upstream service">
        Send a `POST` request to `/api/gateways` with the upstream service's name, endpoint, and any relevant metadata. Include your Bearer token in the `Authorization` header.

        ```bash curl
        curl -X POST https://api.n-gate.io/api/gateways \
          -H "Authorization: Bearer <token>" \
          -H "Content-Type: application/json" \
          -d '{
            "workspaceId": "wsp_abc123",
            "name": "Acme Payment Institution",
            "upstreamUrl": "https://api.acme-payments.com",
            "description": "Primary payment processing partner for US region"
          }'
        ```
      </Step>
      <Step title="Receive the gateway record">
        N-Gate responds with a `gatewayId` that you will reference when registering consumers and constructing proxy calls.

        ```json
        {
          "gatewayId": "gw_pay001",
          "workspaceId": "wsp_abc123",
          "name": "Acme Payment Institution",
          "upstreamUrl": "https://api.acme-payments.com",
          "createdAt": "2024-01-15T12:00:00Z"
        }
        ```
      </Step>
    </Steps>
  </Tab>

  <Tab title="List Gateways">
    Retrieve all gateways registered in your workspace. Bearer authentication is required.

    ```bash curl
    curl -X GET "https://api.n-gate.io/api/gateways?workspaceId=wsp_abc123" \
      -H "Authorization: Bearer <token>"
    ```

    Use this endpoint to audit active upstream connections, verify endpoint URLs, and confirm which services are currently reachable through N-Gate.

    <Tip>
      Regularly audit your gateway list to remove connections to decommissioned upstream services. Inactive gateway records are an unnecessary attack surface.
    </Tip>
  </Tab>

  <Tab title="Delete a Gateway">
    Remove a gateway when the upstream service is decommissioned or the integration is no longer needed. Bearer authentication is required.

    ```bash curl
    curl -X DELETE https://api.n-gate.io/api/gateways/{gatewayId} \
      -H "Authorization: Bearer <token>"
    ```

    Deleting a gateway also invalidates all associated consumer links. Consumers previously authorized through this gateway will receive `403 Forbidden` on their next request until they are re-linked to a different gateway.

    <Warning>
      Gateway deletion is immediate and permanent. Ensure all consumers linked to the gateway have been migrated or notified before removing it to avoid unexpected `403` errors in production traffic.
    </Warning>
  </Tab>
</Tabs>

## Managing Gateway Consumers

<Tabs>
  <Tab title="Register a Consumer">
    <Steps>
      <Step title="Create the consumer record">
        Register the API client by posting its identity and the gateway it should be authorized to use. Bearer authentication is required.

        ```bash curl
        curl -X POST https://api.n-gate.io/api/gateway-consumer \
          -H "Authorization: Bearer <token>" \
          -H "Content-Type: application/json" \
          -d '{
            "workspaceId": "wsp_abc123",
            "gatewayId": "gw_pay001",
            "name": "Mobile Checkout App",
            "description": "iOS/Android checkout client for storefront"
          }'
        ```
      </Step>
      <Step title="Receive the consumer record">
        N-Gate creates the consumer and returns its record, including the assigned `consumerId` you will use to manage or remove this consumer later.

        ```json
        {
          "consumerId": "con_mob001",
          "gatewayId": "gw_pay001",
          "name": "Mobile Checkout App",
          "createdAt": "2024-01-15T12:30:00Z"
        }
        ```
      </Step>
      <Step title="Authorize the consumer">
        With the consumer record created, the linked application can now authenticate with its Bearer token and route requests through the specified gateway.
      </Step>
    </Steps>
  </Tab>

  <Tab title="List Consumers">
    View all consumers registered in your workspace, optionally filtered by gateway. Bearer authentication is required.

    ```bash curl
    curl -X GET "https://api.n-gate.io/api/gateway-consumer?workspaceId=wsp_abc123&gatewayId=gw_pay001" \
      -H "Authorization: Bearer <token>"
    ```

    The response lists each consumer's name, linked gateway, creation date, and status.
  </Tab>

  <Tab title="Remove a Consumer">
    Deauthorize a consumer from a gateway entirely by deleting the consumer record. Bearer authentication is required.

    ```bash curl
    curl -X DELETE https://api.n-gate.io/api/gateway-consumer/{consumerId} \
      -H "Authorization: Bearer <token>"
    ```

    After deletion, any requests using the consumer's credentials are rejected immediately. This is the fastest way to revoke access for an offboarded client application or partner.

    <Tip>
      When offboarding a partner or decommissioning a client application, delete the consumer record as soon as access is no longer needed. Prompt removal limits the window of unauthorized access if the consumer's credentials are compromised.
    </Tip>
  </Tab>
</Tabs>

## How a Request Flows Through a Gateway

<Steps>
  <Step title="Consumer sends request to N-Gate">
    The client application sends an API request to N-Gate's proxy endpoint, authenticating with a valid Bearer token in the `Authorization` header.

    ```bash curl
    curl -X POST https://api.n-gate.io/api/routes/proxy \
      -H "Authorization: Bearer <token>" \
      -H "Content-Type: application/json" \
      -d '{
        "gatewayId": "gw_pay001",
        "path": "/charge",
        "method": "POST",
        "body": { "amount": 1000, "currency": "USD" }
      }'
    ```
  </Step>
  <Step title="N-Gate validates the consumer">
    N-Gate validates the Bearer token, confirms the consumer is registered and active, and verifies the consumer is authorized to use the specified `gatewayId`. If validation fails at any point, N-Gate returns `401 Unauthorized` or `403 Forbidden` — no request leaves N-Gate.
  </Step>
  <Step title="N-Gate resolves the upstream target">
    Using the gateway record linked to the consumer, N-Gate retrieves the upstream service's `upstreamUrl` and constructs the full target endpoint for the request.
  </Step>
  <Step title="Request forwarded to upstream service">
    N-Gate proxies the request to the upstream service, injecting any required service-level credentials or headers. The upstream receives a properly authenticated call without ever seeing the consumer's token.
  </Step>
  <Step title="Response returned to consumer">
    The upstream service's response — status, headers, and body — is streamed back through N-Gate to the consumer application. The consumer sees the upstream's response as if it had called the service directly.
  </Step>
</Steps>

## Use Cases

<Accordion title="Multi-institution connectivity">
  Financial platforms, healthcare networks, and logistics hubs often need to connect to dozens of partner institutions. Create one gateway per institution and manage access centrally — adding or revoking a consumer affects only that specific gateway without disrupting others.
</Accordion>

<Accordion title="Partner API access control">
  Give external partners a Gateway Consumer record scoped only to the gateway they need. Partners never receive raw backend credentials, and you can revoke their access instantly by deleting the consumer record.
</Accordion>

<Accordion title="Isolating environments">
  Create separate gateways for your staging and production upstream services, then link the appropriate consumers to each. Development teams test against staging consumers without any risk of accidentally hitting production.
</Accordion>

<Accordion title="Auditing and visibility">
  Because all traffic flows through N-Gate's gateway layer, you get a single audit trail for every consumer request — what was called, when, by whom, and what the upstream returned. No direct backend calls bypass the log.
</Accordion>
