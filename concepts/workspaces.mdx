A Workspace is the foundational organizational unit in N-Gate. Every resource you create — routes, gateways, remote hand agents, and mandates — lives inside a workspace, and every API call you make is scoped to a specific workspace through the `workspaceId` parameter. By keeping resources fully isolated per workspace, N-Gate gives you a clean multi-tenant boundary between teams, customers, or environments.

## What is a Workspace?

Think of a workspace as a self-contained universe within N-Gate. Nothing inside one workspace is visible to another workspace. Users, routes, gateway connections, and agent registrations are all namespaced to their parent workspace, so two tenants can operate independently without risk of data leakage or configuration collision.

<Note>
  Every authenticated API request (except `/auth/login`) requires a valid `workspaceId`. Make sure you store this value after workspace creation — you will reference it in nearly every subsequent call.
</Note>

### Workspace Hierarchy

The following diagram illustrates the relationship between a workspace, its members, and the resources it contains:

```
┌─────────────────────────────────────────────────────┐
│                    WORKSPACE                        │
│  (workspaceId: wsp_abc123)                          │
│                                                     │
│  ┌─────────────────┐   ┌────────────────────────┐  │
│  │     MEMBERS     │   │       RESOURCES        │  │
│  │                 │   │                        │  │
│  │  User A (admin) │   │  Routes         [n]    │  │
│  │  User B (viewer)│   │  Gateways       [n]    │  │
│  │  User C (editor)│   │  Gateway Consumers [n] │  │
│  └────────┬────────┘   │  Remote Hand Agents [n]│  │
│           │            │  Mandates       [n]    │  │
│           │            └────────────────────────┘  │
│           │                                         │
│  ┌────────▼────────────────────────────────────┐   │
│  │           PERMISSION ROLES                  │   │
│  │  (workspaceId + permissionId + roleId)       │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

One workspace never shares or inherits resources from another — isolation is enforced at the platform level.

## Permission Roles

Access control in N-Gate is governed by a **Permission Role triplet**: `workspaceId + permissionId + roleId`. This three-part key determines exactly what actions a member can perform inside a given workspace.

<CardGroup cols={2}>
  <Card title="Admin" icon="shield-halved">
    Full read/write access across all resources in the workspace. Can invite members, assign roles, and delete the workspace.
  </Card>
  <Card title="Editor" icon="pen-to-square">
    Can create and modify routes, gateways, and consumers, but cannot manage workspace membership or billing.
  </Card>
  <Card title="Viewer" icon="eye">
    Read-only access to all workspace resources. Ideal for auditors, monitoring dashboards, or stakeholder accounts.
  </Card>
  <Card title="Custom Role" icon="sliders">
    Fine-grained roles scoped to specific permission IDs let you grant minimal privilege for service accounts and integrations.
  </Card>
</CardGroup>

<Tip>
  Follow the principle of least privilege: assign service accounts the narrowest role that allows them to complete their task. Use viewer roles for any account that only needs to read configuration data.
</Tip>

## Common Workspace Operations

<Tabs>
  <Tab title="Create a Workspace">
    <Steps>
      <Step title="Authenticate">
        Obtain a Bearer token by posting your credentials to `/auth/login`. All subsequent requests must include this token in the `Authorization` header.

        ```http
        POST https://api.n-gate.io/auth/login
        Content-Type: application/json

        {
          "email": "you@example.com",
          "password": "your-password"
        }
        ```
      </Step>
      <Step title="Create the Workspace">
        Send a `POST` request to `/api/workspaces` with a unique name and optional metadata.

        ```http
        POST https://api.n-gate.io/api/workspaces
        Authorization: Bearer <token>
        Content-Type: application/json

        {
          "name": "production-tenant",
          "description": "Live production environment for Acme Corp"
        }
        ```
      </Step>
      <Step title="Store the workspaceId">
        The response body includes the `workspaceId`. Save this value — you will pass it as a path parameter or request field in every subsequent API call.

        ```json
        {
          "workspaceId": "wsp_abc123",
          "name": "production-tenant",
          "createdAt": "2024-01-15T10:30:00Z"
        }
        ```
      </Step>
    </Steps>
  </Tab>

  <Tab title="List Workspaces">
    Retrieve all workspaces your account belongs to with a single `GET` request.

    ```http
    GET https://api.n-gate.io/api/workspaces
    Authorization: Bearer <token>
    ```

    The response returns an array of workspace objects, each containing the `workspaceId`, name, your role within that workspace, and timestamps.

    <Note>
      The list is filtered to workspaces where your account holds at least one Permission Role. Workspaces you are not a member of are never returned.
    </Note>
  </Tab>

  <Tab title="Invite Members">
    <Steps>
      <Step title="Send the Invitation">
        Post to the workspace members endpoint, specifying the invitee's email and the role they should receive.

        ```http
        POST https://api.n-gate.io/api/workspaces/{workspaceId}/members
        Authorization: Bearer <token>
        Content-Type: application/json

        {
          "email": "colleague@example.com",
          "roleId": "role_viewer"
        }
        ```
      </Step>
      <Step title="Confirm the Invitation">
        The invitee receives an email with a confirmation link. Once they accept, their account is linked to the workspace with the assigned role. You can list pending invitations at any time using `GET /api/workspaces/{workspaceId}/invitations`.
      </Step>
    </Steps>

    <Warning>
      Only workspace **admins** can invite new members or modify existing roles. Attempting this action with a non-admin token returns a `403 Forbidden` response.
    </Warning>
  </Tab>

  <Tab title="Assign Roles">
    Update a member's role by sending a `PATCH` request to their membership record.

    ```http
    PATCH https://api.n-gate.io/api/workspaces/{workspaceId}/members/{memberId}
    Authorization: Bearer <token>
    Content-Type: application/json

    {
      "roleId": "role_editor"
    }
    ```

    N-Gate immediately applies the updated Permission Role triplet (`workspaceId` + `permissionId` + `roleId`). Any in-flight tokens the member holds will reflect the new permissions on their next request.
  </Tab>
</Tabs>

## Workspace Isolation

N-Gate enforces strict data isolation between workspaces at every layer of the stack:

<Accordion title="API-level isolation">
  Every resource endpoint is prefixed or filtered by `workspaceId`. Even if two tenants create routes with identical paths, those routes exist independently and are never co-mingled in query results.
</Accordion>

<Accordion title="Token-scoped access">
  Bearer tokens are issued in the context of a user account and validated against that user's Permission Role triplets. A token cannot be used to access a workspace the user has not been invited to, regardless of how the request is constructed.
</Accordion>

<Accordion title="Audit and visibility boundaries">
  Logs, metrics, and audit trails are partitioned per workspace. Your team sees only the activity generated within your own workspace.
</Accordion>
