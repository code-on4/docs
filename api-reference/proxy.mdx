The Proxy API is the runtime execution layer of N-Gate. While the Routes API lets you define _where_ traffic should go, the Proxy API is what actually moves data. When you call `POST /api/routes/proxy`, N-Gate looks up the route by its `routeId`, enriches the outbound request with any additional headers you supply, and forwards the `payload` to the upstream `target` URL defined on that route. The upstream response — status code, headers, and body — is then proxied back to your caller verbatim, making N-Gate transparent to both the sender and the downstream service.

---

## Forward a Request via Proxy

<br />

`POST /api/routes/proxy`

Dispatches a request through a registered route to its configured upstream target and returns the upstream response directly to the caller.

### How Proxying Works

When N-Gate receives a proxy request it performs the following steps in order:

1. **Route resolution** — The `routeId` is looked up in the database. If no matching route is found, N-Gate returns `404` before contacting any upstream.
2. **Header assembly** — N-Gate merges any `headers` you provide in the request body with a set of standard forwarding headers (e.g. `X-Forwarded-For`, `X-N-Gate-Route-ID`). Your headers take precedence in the case of key conflicts.
3. **Upstream dispatch** — The `payload` is serialized as a JSON body and sent to the route's `target` URL using the method defined on the route.
4. **Response relay** — The upstream HTTP status, response headers, and body are returned to the caller unchanged. N-Gate does not modify or buffer the upstream response body.

<Note>
  N-Gate injects `X-Forwarded-For` and `X-N-Gate-Route-ID` headers into every proxied request for traceability. These cannot be overridden via the `headers` field.
</Note>

### Request Body

<ParamField body="routeId" type="string" required>
  The unique identifier of the route to proxy through. Must correspond to an existing, active route. Obtain route IDs from the [Routes API](/api-reference/routes).
</ParamField>

<ParamField body="payload" type="object">
  An optional JSON object to forward as the request body to the upstream service. The structure is entirely defined by your upstream API — N-Gate passes this through without transformation. Omit this field for routes that do not require a request body (e.g. `GET` routes).
</ParamField>

<ParamField body="headers" type="object">
  An optional map of additional HTTP headers to include in the upstream request. Useful for passing correlation IDs, API keys, or other metadata required by the upstream service.

  ```json
  {
    "X-Correlation-ID": "req-789",
    "X-Custom-Token": "sk_live_abc"
  }
  ```

  <Note>
    Values must be strings. Headers that conflict with N-Gate's reserved forwarding headers (`X-Forwarded-For`, `X-N-Gate-Route-ID`) are silently ignored.
  </Note>
</ParamField>

---

### Response

The proxy endpoint returns the upstream service's response verbatim. The HTTP status code, response headers, and body are all relayed directly from the upstream. N-Gate does not impose its own response envelope on successful proxy calls.

<ResponseField name="(upstream response body)" type="any">
  The raw response body from the upstream service. This is returned exactly as received — it may be a JSON object, a string, or any other content type depending on the upstream API.
</ResponseField>

<Warning>
  N-Gate returns its own structured error only when the failure originates **within N-Gate itself** (e.g. route not found, gateway timeout). If the upstream service returns a `4xx` or `5xx`, that status code and body are passed back to you as-is without wrapping.
</Warning>

---

### N-Gate Gateway Errors

The following errors are generated by N-Gate, not the upstream service:

<ResponseField name="error" type="string">
  A human-readable description of what went wrong at the gateway level.
</ResponseField>

<ResponseField name="statusCode" type="integer">
  The HTTP status code of the N-Gate error. See the table below for possible values.
</ResponseField>

<ResponseField name="timestamp" type="string (ISO 8601)">
  The UTC timestamp at which the error occurred in N-Gate.
</ResponseField>

| Status Code | Cause                                                                                  |
|-------------|----------------------------------------------------------------------------------------|
| `400`       | Missing `routeId` in the request body                                                  |
| `401`       | Missing or invalid `Authorization` token                                               |
| `403`       | Authenticated user does not have access to the workspace this route belongs to         |
| `404`       | No route found matching the provided `routeId`                                         |
| `502`       | N-Gate reached the upstream target but received an invalid or no response (Bad Gateway)|
| `504`       | The upstream service did not respond within N-Gate's timeout window (Gateway Timeout)  |

### Example

<CodeGroup>
```bash cURL
curl --request POST \
  --url "https://api.n-gate.io/api/routes/proxy" \
  --header "Authorization: Bearer <token>" \
  --header "Content-Type: application/json" \
  --data '{
    "routeId": "route_abc123",
    "payload": {
      "amount": 500,
      "currency": "USD"
    },
    "headers": {
      "X-Correlation-ID": "req-789"
    }
  }'
```

```json Upstream Success (200)
{
  "transactionId": "txn_0099aabbcc",
  "status": "approved",
  "amount": 500,
  "currency": "USD",
  "processedAt": "2024-06-01T12:01:30Z"
}
```

```json N-Gate Error — Route Not Found (404)
{
  "error": "Route not found",
  "statusCode": 404,
  "timestamp": "2024-06-01T12:00:00Z"
}
```

```json N-Gate Error — Upstream Unreachable (502)
{
  "error": "Bad gateway: upstream service did not respond",
  "statusCode": 502,
  "timestamp": "2024-06-01T12:00:05Z"
}
```
</CodeGroup>

## Header Forwarding Reference

N-Gate automatically injects the following headers into every proxied request, regardless of what is passed in the `headers` field:

| Header               | Value                                                      |
|----------------------|------------------------------------------------------------|
| `X-Forwarded-For`    | The IP address of the original caller                      |
| `X-N-Gate-Route-ID`  | The `routeId` used for this proxy request                  |
| `Content-Type`       | `application/json` (always set for the proxied body)       |

Any headers provided in the `headers` request field are merged on top of these defaults. The two reserved headers above (`X-Forwarded-For`, `X-N-Gate-Route-ID`) cannot be overridden.
