This guide walks you through the four essential steps to start using N-Gate: obtaining a Bearer token, creating a Workspace, listing your routes, and sending a proxied request to an upstream service. By the end, you'll have a working end-to-end flow you can build on for your own integrations.

<Note>
  All requests in this guide use `https://api.n-gate.io` as the base URL. Make sure you have valid credentials before starting — contact your platform administrator if you don't have an account yet.
</Note>

---

<Steps>

  <Step title="Authenticate and Get Your Bearer Token">

    Before you can call any N-Gate endpoint, you need a Bearer token. Send your credentials to `POST /auth/login` — this is the only endpoint that does **not** require an `Authorization` header.

    <CodeGroup>

    ```bash curl
    curl -X POST https://api.n-gate.io/auth/login \
      -H "Content-Type: application/json" \
      -d '{
        "email": "admin@example.com",
        "password": "your-password"
      }'
    ```

    </CodeGroup>

    **Request body**

    | Field      | Type   | Required | Description                        |
    |------------|--------|----------|------------------------------------|
    | `email`    | string | ✓        | The email address for your account |
    | `password` | string | ✓        | Your account password              |

    **Example response**

    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "user": {
        "id": "usr_abc123",
        "email": "admin@example.com"
      }
    }
    ```

    Copy the value of `token` — you'll include it as a `Bearer` token in the `Authorization` header for every subsequent request:

    ```text
    Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
    ```

    <Warning>
      Treat your token like a password. Never log it, commit it to source control, or pass it as a URL query parameter. See the [Authentication guide](/authentication) for storage best practices.
    </Warning>

  </Step>

  <Step title="Create a Workspace">

    A Workspace is the container for all your routes, gateways, and consumers. Create one by calling `POST /api/workspaces` with a name and optional description. Replace `<YOUR_TOKEN>` with the token you received in Step 1.

    <CodeGroup>

    ```bash curl
    curl -X POST https://api.n-gate.io/api/workspaces \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <YOUR_TOKEN>" \
      -d '{
        "name": "My Institution",
        "description": "Primary workspace"
      }'
    ```

    </CodeGroup>

    **Request body**

    | Field         | Type   | Required | Description                               |
    |---------------|--------|----------|-------------------------------------------|
    | `name`        | string | ✓        | A human-readable name for the Workspace   |
    | `description` | string |          | An optional description of the Workspace  |

    **Example response**

    ```json
    {
      "id": "ws_abc123",
      "name": "My Institution",
      "description": "Primary workspace",
      "createdAt": "2024-01-15T10:00:00Z"
    }
    ```

    <Tip>
      Save the returned `id` (e.g., `ws_abc123`). You'll reference it when creating routes and gateways scoped to this Workspace.
    </Tip>

  </Step>

  <Step title="Create and List Your Routes">

    Routes map an inbound API path to an upstream target service. First, create a route using `POST /api/routes`, then verify it exists by calling `GET /api/routes`.

    **Create a route**

    <CodeGroup>

    ```bash curl
    curl -X POST https://api.n-gate.io/api/routes \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <YOUR_TOKEN>" \
      -d '{
        "path": "/api/v1/payments",
        "target": "https://payments.example.com/v1",
        "method": "POST"
      }'
    ```

    </CodeGroup>

    **Request body**

    | Field    | Type   | Required | Description                                                  |
    |----------|--------|----------|--------------------------------------------------------------|
    | `path`   | string | ✓        | The inbound path N-Gate will match against incoming requests |
    | `target` | string | ✓        | The full URL of the upstream service to forward requests to  |
    | `method` | string | ✓        | The HTTP method this route handles (e.g., `GET`, `POST`)     |

    **Example response**

    ```json
    {
      "id": "route_xyz789",
      "path": "/api/v1/payments",
      "target": "https://payments.example.com/v1",
      "method": "POST",
      "workspaceId": "ws_abc123",
      "createdAt": "2024-01-15T10:05:00Z"
    }
    ```

    **List all routes**

    Once created, you can retrieve all routes in your Workspace:

    <CodeGroup>

    ```bash curl
    curl -X GET https://api.n-gate.io/api/routes \
      -H "Authorization: Bearer <YOUR_TOKEN>"
    ```

    </CodeGroup>

    **Example response**

    ```json
    {
      "routes": [
        {
          "id": "route_xyz789",
          "path": "/api/v1/payments",
          "target": "https://payments.example.com/v1",
          "method": "POST",
          "workspaceId": "ws_abc123",
          "createdAt": "2024-01-15T10:05:00Z"
        }
      ],
      "total": 1
    }
    ```

    <Note>
      Copy the `id` field from the created route (e.g., `route_xyz789`). You'll need it in the next step to send a proxied request.
    </Note>

  </Step>

  <Step title="Make Your First Proxied Request">

    With a route registered, you can now send a proxied request through N-Gate using `POST /api/routes/proxy`. N-Gate will authenticate the request, match it to the route, and forward the `payload` to your upstream target — keeping the internal target URL hidden from the caller.

    <CodeGroup>

    ```bash curl
    curl -X POST https://api.n-gate.io/api/routes/proxy \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer <YOUR_TOKEN>" \
      -d '{
        "routeId": "route_xyz789",
        "payload": {
          "amount": 100
        }
      }'
    ```

    </CodeGroup>

    **Request body**

    | Field     | Type   | Required | Description                                                         |
    |-----------|--------|----------|---------------------------------------------------------------------|
    | `routeId` | string | ✓        | The `id` of the route to proxy this request through                 |
    | `payload` | object | ✓        | The request body to forward to the upstream service                 |

    **Example response**

    The response body is passed through directly from your upstream service. For example:

    ```json
    {
      "status": "success",
      "transactionId": "txn_001abc",
      "amount": 100,
      "currency": "USD",
      "processedAt": "2024-01-15T10:10:00Z"
    }
    ```

    <Tip>
      If the proxy request fails with a `404`, double-check that the `routeId` matches the `id` returned when you created the route, and that the route belongs to the same Workspace your token is scoped to.
    </Tip>

  </Step>

</Steps>
